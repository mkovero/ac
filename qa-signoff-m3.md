## correctness issues

1. **Found and fixed during the build itself** (listed for completeness, not new findings by this review): the `FreqRange::zoom` log-arithmetic violation (AC1) and the `ac-scene::ticks` NaN-on-degenerate-input gap (AC6) — both caught by the developer's own enforcement tests/manual review during implementation, not by this QA pass. This review's job was to independently confirm both fixes are real and complete, which is done above (re-ran the AC1 scan after the fix, hand-verified the NaN mechanism and the guard's placement).
2. **New in this review**: both live-integration tests (AC3, AC4) shipped with tolerance bounds set by feel (1.0 dB, 0.2 dB) rather than measurement — a direct instance of the lesson M1.5 established as standing policy. Re-measured both with an artificially tight bound first, tightened both to the real observed behavior (0.01 dB and 0.1 dB respectively), with the actual measured numbers now in the test comments. Not a correctness bug (the loose bounds would still have caught a real regression), but exactly the class of gap this project's QA discipline exists to close before merge, not after.
3. No other correctness issues found on a full line-by-line read of every new file in `crates/ac-view/src/`.

## gate check — A3 pause (qa.md pending-gate clauses), and adequacy ruling

The architect review's decision 2 already corrected the handoff's original "lavapipe, known limitation" framing (which this review independently re-confirmed was unsupported by any prior repo history when first raised). Beyond accepting that correction, **this review went further and actually attempted the manual real-adapter run the handoff proposes as non-CI evidence**, rather than deferring it:

- Built `ac-view` in release mode, spawned a real `ac-daemon --fake-audio` instance, and ran the actual `ac-view` binary against a real Xvfb X11 display (`:98`, `Xvfb :98 -screen 0 1024x768x24`) for 8 seconds under `timeout`.
- Result: **no crash, no panic, empty stderr/stdout** — the process was still running when the timeout killed it (exit 124), which is success, not failure, for this check.
- Important clarification this run surfaces: `ac-view` uses `eframe`'s `glow` (OpenGL) backend, not `wgpu`. "Lavapipe" is specifically Mesa's *Vulkan* software rasterizer — `wgpu`'s concern, not `glow`'s. Whatever the historical basis for the handoff's original lavapipe worry was (still uncorroborated in this repo), it doesn't describe a risk `ac-view`'s actual chosen backend (decision 3, architect review) is exposed to. The successful Xvfb run used Mesa's OpenGL software path (`llvmpipe`), a different component entirely.
- No screenshot-capture tool (`import`/`xwd`/`scrot`) was available in this sandbox to produce a literal pixel artifact for the PR — this review could confirm the app runs and paints without crashing under a real X server, but not attach an image. That's a real, disclosed limitation of *this specific environment*, not of the approach; a developer with a normal desktop environment can trivially get the screenshot this review couldn't.

**Ruling**: the shape-level geometry test (AC2, mutation-verified) is adequate as the CI-blocking gate. The manual real-adapter run is no longer merely proposed — it was performed, successfully, in this review, and the backend-mismatch point (glow vs. the wgpu-specific lavapipe concern) should be added to `handoff-ac-view.md` or `SNAPSHOT.md`-adjacent docs before M4 revisits this, so the next reviewer doesn't inherit an inapplicable worry. A literal screenshot for the PR is still outstanding (environment-tooling gap, not a correctness question) and shouldn't block this merge — it's the pixel-level evidence tier the handoff already designated non-blocking.
